<summary>🧭 How to read this report (framing & method)</summary>

**What this is ultimately measuring.** When you hand work to an AI here, does it behave like a brand-new hire or like an engineer who's been in the org eighteen months? The difference isn't capability - it's *externalized context*: knowing where things live, which contracts are load-bearing, where the minefields are. An AI contributor is structurally always the new hire - empty head each session, one narrow slice through its context window - so this report scores how much of the tenured engineer's implicit map the codebase has made explicit and navigable. The aim is **legibility you can trust, not omniscience you can't.** The write-side layers aren't a leash on the agent - they're the guardrails that make a costly mistake hard to make for any contributor, human or AI.

This is an improvement roadmap, not a verdict. It pairs three views: where the codebase is today (the complexity heatmap), whether an agent can navigate it (the doc graph), and what keeps it from getting worse (the 0-8 score). **A repo can be 8/8 and still carry real structural pain** - that is exactly this run: complete scaffolding over a core that is accreting. The views matter together.

**The legacy-transition lens.** The hard part of AI-assisted coding is the same one the industry has faced for 25 years: what happens when the people who understood the code are gone, or the code outruns anyone's ability to understand it? AI only changes the *velocity* - and the keyhole findings here name the mechanism: an agent does what is asked, what is asked is feature after feature, and nothing in that loop ever asks for a refactor, so files only grow (`accretion_ratchet`) and promises go unkept (`unactioned_intent`). Read the actions through legacy-code discipline:

- **"Legacy code is code without tests."** (Feathers, 2004) - here the core *has* tests, so it is safe to refactor down behind them. Extract and split; don't rewrite.
- **Don't rewrite - strangle.** (Spolsky 2000; Fowler's Strangler Fig) - the agent instinct to regenerate `assess_core.py` from intent would discard accreted fixes. Replace incrementally behind the existing tests.
- **Hotspots and change-coupling are the map.** (Tornhill / CodeScene) - the heatmap and the `hidden_coupling` finding *are* hotspot and change-coupling analysis; the coupling here is already documented as a seam.
- **Decide per component (the 7 Rs).** Not everything flagged should be refactored - retain, retire, rehost, replatform, refactor, re-architect, rebuild.

**How it's measured.** Almost entirely traditional tooling - static analysis, git history, graph metrics over docs and code. The model only writes the prose around those numbers; it does no scanning. Fast, near-zero in model tokens, reproducible run-to-run.
